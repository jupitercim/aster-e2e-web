# AsterDEX E2E 自动化测试代码规范

本文档从现有用例中提炼，所有新增 spec 文件必须遵守。

---

## 1. 文件命名

- 使用 **kebab-case**，以 `.spec.ts` 结尾
- 按功能模块命名，与测试计划保持一致

```
✅ future-limit-order.spec.ts
✅ spot-order.spec.ts
❌ futureOrder.spec.ts
❌ future_order.spec.ts
```

---

## 2. 文件结构模板

每个 spec 文件必须按以下顺序组织：

```typescript
import { test, expect } from '../fixtures/auth';

// 1. URL 辅助函数（如有）
function getXxxUrl(): string {
  const base = process.env.EXCHANGE_URL || '';
  const origin = new URL(base).origin;
  return `${origin}/zh-CN/your-path`;
}

test.describe.serial('AsterDEX - 模块名称', () => {

  // 2. 跨测试共享变量（如有）
  let someValue: number = 0;

  // 3. 辅助函数（handleConfirmDialog、checkToast 等）
  async function handleConfirmDialog(page: any) { ... }
  async function checkToast(page: any, keywords: string[], label: string) { ... }

  // 4. 测试用例（按业务流程顺序排列）
  test('描述 test 1', async ({ loggedInPage: page }) => { ... });
  test('描述 test 2', async ({ loggedInPage: page }) => { ... });
});
```

---

## 3. 必须使用 `test.describe.serial`

所有 spec 文件必须使用串行模式，原因：
- MetaMask userdata 目录只能被一个 Chromium 进程占用
- 后续 test 依赖前一个 test 的页面状态

```typescript
✅ test.describe.serial('AsterDEX - 期货限价委托', () => { ... });
❌ test.describe('AsterDEX - 期货限价委托', () => { ... });
```

---

## 4. 导入规范

必须从 `../fixtures/auth` 导入，**不得**直接使用 `@playwright/test`（交易类用例需要已登录的 `loggedInPage`）：

```typescript
✅ import { test, expect } from '../fixtures/auth';
❌ import { test, expect } from '@playwright/test';
```

> 例外：`page-static.spec.ts` 等无需登录的检查可使用 `@playwright/test`。

## 6. 元素定位优先级

按可维护性从高到低：

| 优先级 | 方式 | 示例 |
|--------|------|------|
| 1 | `data-testid` | `page.locator('[data-testid="price-input"]')` |
| 2 | ARIA role | `page.getByRole('button', { name: /买入\|Buy/i })` |
| 3 | Tab/标签文本 | `page.locator('button[role="tab"]:has-text("当前委托")')` |
| 4 | Placeholder | `page.locator('input[placeholder="价格"]')` |
| 5 | 文本内容 | `page.locator('button:text("取消")')` |
| 6 | CSS 选择器 | `page.locator('button[type="submit"]').first()` |

### 多候选文案用循环，禁止硬编码单一文案

```typescript
// ✅ 正确：容错多种文案
const tabNames = ['当前委托', 'Open Orders', '委托', 'Orders'];
for (const name of tabNames) {
  const tab = page.locator(`button[role="tab"]:has-text("${name}")`);
  if (await tab.isVisible({ timeout: 2000 }).catch(() => false)) {
    await tab.click();
    break;
  }
}

// ❌ 错误：单一文案，环境切换后易断
await page.locator('button[role="tab"]:has-text("当前委托")').click();
```

## 7. 标准辅助函数

每个交易类 spec 文件必须包含以下两个辅助函数（直接复制，不要修改签名）：

### 7.1 `handleConfirmDialog`

```typescript
async function handleConfirmDialog(page: any) {
  await page.waitForTimeout(500);

  // 方法1：找"订单确认"弹窗内的确认按钮
  const dialog = page.locator('text=订单确认').locator('..');
  if (await dialog.isVisible({ timeout: 1500 }).catch(() => false)) {
    const dialogBtns = dialog.locator('..').locator('button');
    if (await dialogBtns.count() > 0) {
      await dialogBtns.last().click();
      return;
    }
  }

  // 方法2：找"取消"旁边的确认按钮
  const cancelBtn = page.locator('button:text("取消")');
  if (await cancelBtn.isVisible({ timeout: 1500 }).catch(() => false)) {
    await cancelBtn.locator('..').locator('button').last().click();
    return;
  }

  // 方法3：fallback
  const fallbackBtn = page.getByRole('button', { name: /确认|开多|开空|Confirm/i }).last();
  if (await fallbackBtn.isVisible({ timeout: 1500 }).catch(() => false)) {
    await fallbackBtn.click();
  }
}
```

### 7.2 `checkToast`

```typescript
async function checkToast(page: any, keywords: string[], label: string): Promise<boolean> {
  const selector = keywords.map(kw => `text=${kw}`).join(', ');
  const toast = page.locator(selector).first();
  const appeared = await toast.waitFor({ state: 'visible', timeout: 3000 })
    .then(() => true).catch(() => false);

  if (appeared) {
    const text = await toast.textContent();
    console.log(`[test] ✅ ${label}: ${text?.trim()}`);
    const filename = `test-results/toast-${label.replace(/[^a-zA-Z0-9\u4e00-\u9fa5]/g, '_')}-${Date.now()}.png`;
    await page.screenshot({ path: filename, fullPage: false });
    return true;
  }

  console.log(`[test] ⚠️ 未检测到 ${label} 提示，继续执行`);
  return false;
}
```

## 8. 可选元素处理

对于可能不存在的元素，统一使用 `.catch(() => false)` 模式：

```typescript
// ✅ 正确
const isVisible = await element.isVisible({ timeout: 3000 }).catch(() => false);
if (isVisible) {
  await element.click();
}

// ❌ 错误：未捕获异常，元素不存在时测试直接 crash
if (await element.isVisible({ timeout: 3000 })) {
  await element.click();
}
```

## 9. 日志规范

所有 `console.log` 必须带 `[test]` 前缀，并使用 `✅` / `⚠️` 标识结果：

```typescript
console.log('[test] ✅ 下单成功，订单已出现在委托列表');  // 成功
console.log('[test] ⚠️ 未找到取消按钮，跳过');            // 软跳过
console.warn('[test] ❌ 无法读取 Mark Price');             // 严重错误
```

## 10. 断言规范

### 硬断言 vs 软断言

```typescript
// 硬断言：失败立即终止测试（用于核心功能）
expect(order).toBeVisible({ timeout: 5000 });
expect(isCancelled).toBe(true);

// 软断言：失败记录但继续执行（用于非关键验证）
expect.soft(title).toBeTruthy();
```

### Toast 验证使用 `waitFor`，不用 `isVisible`

```typescript
// ✅ 正确：真正等待元素出现
await toast.waitFor({ state: 'visible', timeout: 3000 });

// ❌ 错误：isVisible 在新版 Playwright 中立即返回，不等待
await toast.isVisible({ timeout: 3000 });
```

---

## 11. 错误处理与跳过

业务上允许不存在的情况，使用 `return` 跳过而非 `throw`：

```typescript
const firstCancelBtn = page.locator('button:text("取消")').first();
if (!(await firstCancelBtn.isVisible({ timeout: 5000 }).catch(() => false))) {
  console.log('[test] ⚠️ 没有委托订单，跳过');
  return;  // ✅ 软跳过
}
```

```typescript
await page.screenshot({
```

## 13. 等待策略

| 场景 | 推荐方式 |
|------|----------|
| 页面加载完成 | `waitForLoadState('domcontentloaded')` |
| 等待网络空闲 | `waitForLoadState('networkidle')` （慎用，超时风险高）|
| 固定等待 | `waitForTimeout(ms)` （尽量缩短，最长 3000ms）|
| 等待元素出现 | `expect(el).toBeVisible({ timeout: 5000 })` |
| 等待 Toast | `toast.waitFor({ state: 'visible', timeout: 3000 })` |

## 14. 并发与 Worker 配置

`playwright.config.ts` 已固定 `workers: 1`，**禁止修改**：

```typescript
// playwright.config.ts
workers: 1,  // MetaMask userdata 目录只能被一个进程占用，必须串行
```

原因：MetaMask 使用 `.metamask-userdata` 持久化目录，多进程并发会触发 `ProcessSingleton` 冲突导致所有测试失败。

## 15. 测试优先级标签（P0–P3）

所有 `test()` 必须通过第二个参数 `{ tag: ['@Px'] }` 标记优先级，**新增用例必须打标，否则不允许合并**。

### 优先级定义

| 标签 | 含义 | 典型场景 |
|------|------|----------|
| `@P0` | 冒烟测试 | 页面能打开、核心下单/交易流程 |
| `@P1` | 高优 | 主要功能验证、关键 UI 可见性 |
| `@P2` | 中优 | 次要功能、边界场景、数据验证 |
| `@P3` | 低优 | 细节验证、提示文案、样式检查 |

### 写法

```typescript
// ✅ 正确：在 test 名称后、async 前加 { tag: [...] }
test('限价挂单 BTC/USDT', { tag: ['@P0'] }, async ({ loggedInPage: page }) => {
  ...
});

// 可同时打多个标签
test('验证隐藏订单 checkbox', { tag: ['@P1', '@smoke'] }, async ({ loggedInPage: page }) => {
  ...
});

// ❌ 错误：缺少 tag
test('限价挂单 BTC/USDT', async ({ loggedInPage: page }) => {
  ...
});
```

### 按级别执行

```bash
# 只跑 P0（冒烟）
npx playwright test --grep "@P0"

# 跑 P0 + P1
npx playwright test --grep "@P0|@P1"

# 排除低优
npx playwright test --grep-invert "@P3"

# 结合环境变量
EXCHANGE_URL=https://asterdex.com/... npx playwright test --grep "@P0"
```

## 16. 环境变量

| 变量 | 说明 | 示例 |
|------|------|------|
| `EXCHANGE_URL` | 期货交易页完整 URL | `https://xxx.finance/zh-CN/trade/pro/futures/BTCUSDT` |
| `TEST_SEED_PHRASE` | MetaMask 助记词（仅测试网）| `word1 word2 ... word12` |
| `METAMASK_PASSWORD` | MetaMask 密码 | `Test1234!` |
| `K8S_CLUSTER` | 灰度环境路由 header | `cluster-a` |

所有用例通过 `process.env.EXCHANGE_URL` 获取基础 URL，其他页面路径从 `origin` 拼接：

```typescript
const origin = new URL(process.env.EXCHANGE_URL!).origin;
const targetUrl = `${origin}/zh-CN/portfolio/pro`;
```
